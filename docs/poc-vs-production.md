# Meeting Minutes Agent — PoC 前提と本番ベストプラクティスの差分

> **最終更新日**: 2026-05-11

本リポジトリは **デモ・PoC 用途** で構築されています。本番環境で運用する場合は、以下の差分を確認・対応してください。

---

## 1. セキュリティ

| 項目 | 現状（PoC） | 本番での推奨対応 | 優先度 |
|------|------------|-----------------|--------|
| **認証・認可** | フロントエンド / API ともに認証なし | Azure AD (Entra ID) / Easy Auth による認証を追加。API には OAuth 2.0 Bearer トークン検証を実装 | **高** |
| **CORS** | `allow_origins=["*"]`（全オリジン許可） | フロントエンドの FQDN のみに制限 | **高** |
| **XSRF 保護** | Streamlit の `enableXsrfProtection=false` | `true` に変更、または WAF で保護 | **高** |
| **WAF / API Gateway** | なし（フロントエンドが直接公開） | Azure Front Door + WAF、または API Management を前段に配置 | **高** |
| **ネットワーク** | フロントエンドはパブリック公開 | 必要に応じて IP 制限や Private Link 経由のアクセスに変更 | 中 |
| **コンテナイメージ脆弱性** | スキャンなし | ACR の Microsoft Defender for Containers を有効化し、CI/CD でイメージスキャンを実施 | **高** |
| **ffmpeg** | apt パッケージを直接インストール | 既知の脆弱性がないバージョンを固定し、定期的にアップデート | 中 |
| **シークレット管理** | 環境変数で直接設定 | Azure Key Vault に格納し、Container Apps の Key Vault 参照で注入 | **高** |
| **レート制限** | なし | API Management のレート制限ポリシー、または FastAPI ミドルウェアで実装 | 中 |

---

## 2. 可用性・信頼性

| 項目 | 現状（PoC） | 本番での推奨対応 | 優先度 |
|------|------------|-----------------|--------|
| **ジョブストア** | インメモリ（`dict`）。プロセス再起動で消失 | Redis / Cosmos DB / Azure Queue など永続的なジョブ管理に移行 | **高** |
| **スケーリング** | 単一インスタンス前提（インメモリ状態を共有不可） | ジョブストアの外部化後、水平スケーリングが可能に | **高** |
| **ストレージ冗長性** | `LRS`（ローカル冗長） | 本番は `ZRS`（ゾーン冗長）または `GRS`（地理冗長）を推奨 | **高** |
| **リージョン** | 単一リージョン（japaneast） | DR 要件に応じてマルチリージョン構成を検討 | 中 |
| **ACR SKU** | `Basic`（SLA なし） | `Standard` 以上を推奨。`Premium` で Geo-Replication が利用可能 | 中 |
| **バックアップ** | Blob Storage のバックアップポリシーなし | Blob のバージョニング有効化、または Azure Backup for Blobs を設定 | 中 |
| **Readiness Probe** | なし（Liveness Probe のみ） | Readiness Probe を追加し、起動完了前のリクエスト受付を防止 | 中 |

---

## 3. 運用・監視

| 項目 | 現状（PoC） | 本番での推奨対応 | 優先度 |
|------|------------|-----------------|--------|
| **ログ** | Python `logging` → Log Analytics（保持 30 日） | 構造化ログ（JSON 形式）の導入。Application Insights でのトレース相関 | **高** |
| **アラート** | なし | エラー率・レイテンシ・リソース使用率に対する Azure Monitor アラートを設定 | **高** |
| **CI/CD** | なし（手動ビルド＆デプロイ） | GitHub Actions 等で自動テスト → ACR ビルド → Container Apps デプロイのパイプラインを構築 | **高** |
| **コスト監視** | なし | Azure Cost Management で予算アラートを設定 | 中 |
| **テスト** | ユニットテスト / E2E テストなし | pytest + CI/CD パイプラインでの自動テスト | **高** |
| **ドキュメント** | 本リポジトリ | 運用手順書・障害対応フロー・エスカレーションパスの整備 | 中 |

---

## 4. アプリケーション設計

| 項目 | 現状（PoC） | 本番での推奨対応 | 優先度 |
|------|------------|-----------------|--------|
| **フロントエンド** | Streamlit（プロトタイピング向け） | React / Next.js 等への置き換え。大規模な同時接続に不向き | 中 |
| **音声ファイルサイズ** | 最大 100MB | チャンク分割処理、または Azure Speech Batch Transcription への移行 | 低 |
| **エラーハンドリング** | 基本的なエラーキャッチ | 構造化エラーレスポンス + リトライロジック + Dead Letter Queue | 中 |
| **API バージョニング** | `/api/v1` 固定 | バージョニング戦略の策定（URL / ヘッダー） | 低 |

## 6. 段階的な移行アプローチ

すべてを一度に対応するのではなく、以下の優先順で段階的に対応することを推奨します:

### Phase 1: セキュリティ基盤（1〜2 週間）
- Entra ID 認証
- CORS 制限
- Key Vault 統合
- WAF 配置

### Phase 2: 信頼性向上（2〜3 週間）
- ジョブストアの外部化
- ストレージ冗長性の強化
- CI/CD パイプライン構築

### Phase 3: 可観測性（1〜2 週間）
- Application Insights 統合
- 構造化ログ
- アラート設定

### Phase 4: 最適化（継続的）
- パフォーマンスチューニング
- コスト最適化
- フロントエンドの置き換え検討
